# create-pr

現在のブランチの変更内容を読み取り、レビューしやすい PR タイトルと本文を生成する。

このスキルの役割は「それっぽい文章を作ること」ではなく、
「差分に基づいた事実ベースの説明を、レビュアーに伝わる粒度で整理すること」。

---

## 手順

### 1. 変更内容を把握する

以下のコマンドで現在の状態を確認する:

```bash
# 現在の状態
git status
git log --oneline -10

# 未コミットの差分（作業中の変更）
git diff --stat
git diff

# ベースブランチとの差分（PR 相当の全体差分）
git diff main...HEAD --stat
git diff main...HEAD
```

PR 本文はベースブランチとの差分を主に使う。
未コミットの変更がある場合はそれも含めて把握する。

変更されたファイルの中身を読み、何がどう変わったかを理解する。
コミットメッセージだけに頼らず、実際の差分を確認する。

### 2. RuboCop を実行して自動修正する

```bash
bundle exec rubocop -A
```

- 修正が発生したファイルがあれば、内容を確認してコミットに含める
- 修正後も残るオフェンス（自動修正不可）があればユーザーに報告し、対応を確認してから先に進む
- RuboCop がプロジェクトに導入されていない場合はスキップする

### 3. 変更の性質を分類する

変更を以下のカテゴリに分類して整理する:
- 新機能の追加
- 既存機能の修正・改善
- バグ修正
- リファクタリング
- テストの追加・修正
- 設定・インフラの変更
- DB migration

1つの PR に複数のカテゴリが含まれる場合はそれぞれ区別して書く。

### 4. PR タイトルを作る

タイトルは1案だけ出す。

- 簡潔にする（目安: 50文字以内）
- 変更の目的が伝わる表現にする
- 新機能やバグ修正では結果ベース（「何ができるようになるか / 何が直るか」）、
  リファクタやテスト追加では変更内容ベース（「何をしたか」）が自然。
  変更の種類に応じて適切な表現を選ぶ
- チケット番号やブランチ名から紐づくIDがあれば先頭に付ける

### 5. PR 本文を書く

以下のフォーマットに従う。すべてのセクションを必ず含める。
差分が小さくてもセクションを省略しない。

```markdown
## 概要

（この PR で何をしたか。1〜3行で要約）

## 変更内容

（箇条書きで整理。1項目1意図を意識する。
ファイル名の羅列ではなく、レビュアーに伝わる粒度で説明する。
リファクタ・バグ修正・テスト追加・設定変更などは区別して書く）

## 影響範囲

（この変更が影響するページ、機能、モデル、API などを明示する。
破壊的変更がある場合はここで明確に書く。
DB migration がある場合はここに明記する）

## 動作確認

（確認した手順があれば書く。未確認なら「未確認」と明記する）

## テスト

（追加・変更したテストがあれば書く。テスト未実施なら「未実施」と明記する。
実行コマンドと結果があれば添える）

## レビューで見てほしい点

（設計判断・迷った点・破壊的変更・特定のレビュー観点がある場合に具体的に書く。
なければ「特になし」）

## 未確認・懸念点

（差分だけでは判断できなかった点、動作確認が不十分な箇所、
既知のリスクがあれば書く。なければ「特になし」）
```

### 6. 書く際のルール

- **事実ベースで書く。** 差分から確認できない内容は推測で断定しない。
  変更理由が差分だけでは読み取れない場合、勝手に背景を作らない。
- **不明なことは「未確認」「要確認」と明記する。**
  わからないことを書かないのではなく、わからないと書く。
- **実装されていないこと、差分に存在しないことを書かない。**
- **長文化しすぎない。** レビューしやすさを優先する。
- **ユーザーが特に指定しない限り、日本語で出力する。**

### 7. 最終出力

タイトルと本文をプレビュー表示した後、
「そのまま貼れる PR 文面」としてコードブロックで出力する。

```
**タイトル:** （タイトル）

---

（本文をコードブロックで出力）
```

### 8. PR 作成の確認

プレビュー表示後、以下のように確認する:

```
この内容で `gh pr create` を実行しますか？ (yes/no)
```

- **yes** → `gh pr create` を実行し、作成された PR の URL を表示する
- **no / 修正依頼** → 修正内容をヒアリングして本文を更新し、再度プレビューを表示する

---

## 注意事項

- ブランチの作成・切り替えはしない
- リモートに push されていない場合は `gh pr create` の前に push が必要な旨を伝える
