# tdd-cycle

Rails + RSpec プロジェクトで TDD を1サイクルずつ規律よく進めるためのスキル。

このスキルの役割は「実装を速く終わらせること」ではなく、
「Red → Green → Refactor の順序を守りながら、小さな振る舞いを1つずつ確実に積み上げること」。

---

## Step 1: 今回の対象を1つ決める

会話履歴・git status・最近変更されたファイル・開かれている spec ファイルなどから
現在の作業コンテキストを読み取り、**今回のサイクルで扱う小さい振る舞いを1つだけ**提案する。

**スコープの目安 — 1サイクルに収まる大きさ:**
- モデルの1つのバリデーション
- メソッドの1つの正常ケース or 1つの異常ケース
- コントローラアクションの1つのレスポンス（成功 or 失敗）
- スコープやクラスメソッドの1つの振る舞い

これより大きい場合は実装に入る前に分割案を提示してユーザーの確認を得ること。
ユーザーが確認したら Step 2 へ進む。

### 分割計画の遵守ルール

- **初回実行時**: 作業全体の分割案（番号付きリスト）をユーザーに提示して承認を得る。
  これが以降のサイクルの「計画」になる。
- **2回目以降**: 計画の「次の1つ」をそのまま選ぶ。計画にない項目を勝手に差し込まない。
  計画にある項目を勝手に分割・統合しない。
- **計画変更が必要な場合**: 実装を中断し、「計画を以下のように変更したい。理由は〜」と
  ユーザーに提示して承認を得てから再開する。無断で変更しない。

計画はユーザーとの合意事項であり、「実装上の都合」で一方的に変えない。

---

## Step 2: 🔴 Red — 失敗するテストを書く

対象の振る舞いだけを検証する RSpec の example を1つ（または最小限の数）書く。

- 既存の spec ファイルがあれば追記、なければ適切なパスに新規作成
- Rails プロジェクトの命名規則・ディレクトリ構成に合わせる
  - モデル: `spec/models/`
  - リクエスト: `spec/requests/`
  - システム: `spec/system/`
- `FactoryBot` が使えるなら積極的に使う（ない場合は `Model.new` / `create` で対応）
- テスト対象が存在しない（クラス・メソッドが未定義）でも、先に spec を書く

書いたら「この spec を書きました」とファイルパスと内容を明示する。

---

## Step 3: 🔴 Red — 失敗を確認する

```bash
bundle exec rspec <spec_file_path>
```

失敗出力を確認し、ユーザーに見せる。

**失敗が確認できるまで Step 4 に進まない。**

### テストが意図せずパスした場合

2つのケースを区別する:

**A. テストが対象の振る舞いを正しく検証できていない場合:**
テスト自体を見直し、対象の振る舞いが未実装なのにパスする原因を特定して修正する。
修正後に改めて Red を確認する。

**B. 前のサイクルの実装で既に正当にカバーされている場合:**
- そのテストは回帰テスト（リグレッションガード）として意味がある
- ユーザーに「このテストは前のサイクルの実装で既にパスしています。
  回帰テストとして残しますか？」と確認する
- 残す場合: 実装なし（Green 既達）のサイクルとして短縮完了し、Step 7 → Step 8 へ進む
- 不要な場合: テストを削除してサイクルをスキップする

---

## Step 4: 🟢 Green — 最小限の実装をする

テストをパスさせるための**最小限の**コードを書く。

- きれいなコードより「通ること」を優先する（Refactor は次のステップ）
- 対象の振る舞い以外のコードを書かない
- 将来使いそうなコードを「ついでに」入れない
- 変更が1ファイル・数行に収まらない場合は立ち止まり、さらに分割できないか考える

### 計画外の変更が必要になった場合

テストを通すために、今回の計画項目に含まれないファイルや振る舞いの変更が必要になることがある。
その場合は実装を止めて、以下を確認する:

1. **計画の順序が間違っていないか？**
   依存関係を見直し、先にやるべき項目が他にあるなら計画の順序変更をユーザーに提案する。
2. **順序は正しいが追加変更が必要な場合:**
   「テストを通すために〈追加変更の内容〉も必要です。この変更を含めてよいですか？」と
   ユーザーに確認してから進める。

「テストを通すために必要だから」を理由に、計画外の変更を無断で行わない。

---

## Step 5: 🟢 Green — 成功を確認する

```bash
bundle exec rspec <spec_file_path>
```

パスを確認し、出力をユーザーに見せる。

**パスが確認できるまで Step 6 に進まない。**

---

## Step 6: 🔵 Refactor — 必要なら整理する

実装コードと spec の両方を見直し、読みやすくなる箇所があれば整理する。
ただしリファクタリングの対象は**今回触ったコードだけ**。他の箇所に手を広げない。

整理したら再度テストを実行してグリーンを維持していることを確認する。

リファクタリングが不要であれば、そのまま Step 7 へ進む。

---

## Step 7: 関連テストを実行する

今回の変更が他の既存テストを壊していないことを確認する。

```bash
# 変更したファイルに応じて適切なスコープで実行する
bundle exec rspec spec/models/         # モデルを触った場合
bundle exec rspec spec/requests/       # コントローラを触った場合
bundle exec rspec spec/               # 全体（速い場合）
```

失敗があれば修正し、グリーンを維持してから終了する。

---

## Step 8: サイクル終了 — 必ず以下を報告する

```
---
🏁 今回のサイクル完了

【今回対象にした振る舞い】
（例: Visitor モデルで visitor_token の presence バリデーションが効くこと）

【完了したこと】
- spec ファイル: （パス）に example を追加
- 実装: （ファイルパス）に（何を）追加

【残っていること】
（計画のうち、まだ完了していない項目を番号付きで列挙）

【作業全体の状態】
未完了 / 完了

【次の小さい候補】
（計画の次の項目を示す）

まだ残りがあります。続けるには /tdd-cycle を再実行してください。
---
```

作業全体が完了した場合のみ「完了」と記載し、それ以外は「未完了」とする。

---

## 注意事項

- **ブランチ作成・切り替えはしない**（すでに正しいブランチにいる前提）
- **Issue 作成はしない**
- **引数は受け取らない**（コンテキストから対象を判断する）
- 1サイクルで Issue 全体を終わらせようとしない
- Red を見ずに Green に進まない、Green を見ずに Refactor に進まない — この順序は速度より優先する
- 承認済みの計画を無断で変更しない
