# tdd-cycle

Rails + RSpec プロジェクトで TDD を1サイクルずつ規律よく進めるためのスキル。

このスキルの役割は「実装を速く終わらせること」ではなく、
「Red → Green → Refactor の順序を守りながら、小さな振る舞いを1つずつ確実に積み上げること」。

---

## Step 1: 今回の対象を1つ決める

会話履歴・git status・最近変更されたファイル・開かれている spec ファイルなどから
現在の作業コンテキストを読み取り、**今回のサイクルで扱う小さい振る舞いを1つだけ**提案する。

### 振る舞いベースで分割する

分割の単位は「外から観測できる振る舞い」にする。
実装の内部構造（どのバリデーションを足す、どのクラスを継承する、migration のデフォルト値）で
切り分けない。

**良い分割の例 — 「Visitor の cookie 識別」を分割する場合:**
1. visitor_token がない Visitor は無効
2. 重複した visitor_token は無効
3. 初回 GET / で Visitor が1件作成される
4. 初回 GET / で visitor_token cookie が設定される
5. 同じ cookie で再アクセスしても Visitor は増えない
6. 同じ cookie で再アクセスすると last_visited_at が更新される
7. cookie をクリアして再アクセスすると別 Visitor になる

それぞれが「〜すると〜になる」という形で表現でき、テストのタイトルにそのまま使える。

**良くない分割の例 — 同じ作業を内部実装名で切った場合:**
1. visitor_token の presence バリデーション
2. visitor_token の uniqueness バリデーション
3. first_visited_at / last_visited_at の presence バリデーション
4. score のデフォルト値（0）
5. PublicController の実装
6. HomeController / ArticlesController の継承変更

何が不十分か:
- 1, 2 は振る舞いとしては正しいが、内部実装名のまま書かれている。
  「visitor_token がない Visitor は無効」「重複した visitor_token は無効」のように
  振る舞いの表現に言い換えるべき。
- 「first_visited_at の presence バリデーション」はチケットの受け入れ条件に直接対応しない内部制約。
  それが必要かどうかは「初回 GET / で Visitor が作成される」の実装過程で判断すればよい。
- 「score のデフォルト値」はこのチケットの受け入れ条件ではなく、別チケット（スコアリング）の関心事かもしれない。
- 「PublicController の実装」「継承変更」は手段であり、振る舞いではない。

### 分割の判断基準

各項目が以下を満たしているか確認する:
- **「〜すると〜になる」の形で表現できるか？** — できなければ実装の手段を書いている可能性が高い。
- **RSpec の `it` のタイトルにそのまま使えるか？** — 使えなければ粒度がおかしい。
- **そのチケットの受け入れ条件やテスト観点に対応しているか？** — 対応しない項目は
  本当にこのチケットで必要か疑う。

内部的な実装（バリデーション追加、migration 変更、クラス継承変更など）は
通常、振る舞いを実現するための手段として Green フェーズで自然に生まれるもの。
ただし、そのチケットの受け入れ条件や観測可能な振る舞いに直接結びついている場合は
サイクルの対象にしてよい。
例: 「visitor_token がない Visitor は無効」はバリデーションの話だが、
データ整合性として外から観測できる振る舞いなので対象になる。

### 分割計画の遵守ルール

- **初回実行時**: 作業全体の分割案（番号付きリスト）をユーザーに提示して承認を得る。
  これが以降のサイクルの「計画」になる。
- **2回目以降**: 計画の「次の1つ」をそのまま選ぶ。計画にない項目を勝手に差し込まない。
  計画にある項目を勝手に分割・統合しない。
- **計画変更が必要な場合**: 実装を中断し、「計画を以下のように変更したい。理由は〜」と
  ユーザーに提示して承認を得てから再開する。無断で変更しない。

計画はユーザーとの合意事項であり、「実装上の都合」で一方的に変えない。

---

## Step 2: 🔴 Red — 失敗するテストを書く

対象の振る舞いだけを検証する RSpec の example を1つ（または最小限の数）書く。

- 既存の spec ファイルがあれば追記、なければ適切なパスに新規作成
- Rails プロジェクトの命名規則・ディレクトリ構成に合わせる
  - モデル: `spec/models/`
  - リクエスト: `spec/requests/`
  - システム: `spec/system/`
- `FactoryBot` が使えるなら積極的に使う（ない場合は `Model.new` / `create` で対応）
- テスト対象が存在しない（クラス・メソッドが未定義）でも、先に spec を書く

書いたら「この spec を書きました」とファイルパスと内容を明示する。

---

## Step 3: 🔴 Red — 意図した失敗を確認する

```bash
bundle exec rspec <spec_file_path>
```

失敗出力を確認し、ユーザーに見せる。

**失敗が確認できるまで Step 4 に進まない。**

### 失敗の質を確認する

テストが失敗しても、すべてが有効な Red とは限らない。
失敗が「対象の振る舞いが未実装であることを示しているか」を判断する。

**Red とみなしてよい失敗:**
- 期待値と実際の値が食い違っている（`expected X, got Y`）
- 対象の振る舞いに直接対応するクラスやメソッドが未定義（`NameError: uninitialized constant Visitor`、
  `NoMethodError: undefined method 'recalculate_score!'` など）
  — TDD の初期サイクルではこれが自然な Red になる

**Red とみなさない失敗（前提の不備）:**
- テスト対象と無関係なクラスの `LoadError` / `NameError`
- 構文エラー（`SyntaxError`）
- Factory / fixture の不備による失敗
- DB マイグレーション未実行によるテーブル不存在
- 環境設定漏れ（gem 未インストール、設定ファイル不足など）

後者の場合は Step 2 に戻って前提を整えてから、改めてテストを実行する。

### テストが意図せずパスした場合

2つのケースを区別する:

**A. テストが対象の振る舞いを正しく検証できていない場合:**
テスト自体を見直し、対象の振る舞いが未実装なのにパスする原因を特定して修正する。
修正後に改めて Red を確認する。

**B. 前のサイクルの実装で既に正当にカバーされている場合:**
- そのテストは回帰テスト（リグレッションガード）として意味がある
- ユーザーに「このテストは前のサイクルの実装で既にパスしています。
  回帰テストとして残しますか？」と確認する
- 残す場合: 実装なし（Green 既達）のサイクルとして短縮完了し、Step 7 → Step 8 へ進む
- 不要な場合: テストを削除してサイクルをスキップする

---

## Step 4: 🟢 Green — 最小限の実装をする

テストをパスさせるための**最小限の**コードを書く。

- きれいなコードより「通ること」を優先する（Refactor は次のステップ）
- 対象の振る舞い以外のコードを書かない
- 将来使いそうなコードを「ついでに」入れない
- 変更が1ファイル・数行に収まらない場合は立ち止まり、さらに分割できないか考える

### ついで実装の防止

実装中に「今のコードを少し拡張すれば、計画の別の項目も同時に満たせる」と気づくことがある。
その場合でも、今回の対象以外の振る舞いを満たす実装は追加しない。

- 今回のテストを通す最小限のコードだけを書く
- 別の振る舞いもカバーできそうだと気づいた場合は、それを「次サイクル候補メモ」として
  Step 8 の報告に記録するだけにとどめる
- 例: 「初回 GET / で Visitor が作成される」のサイクル中に cookie 設定や
  再訪問時の last_visited_at 更新ロジックまで一気に書かない。
  cookie 設定は次のサイクル「初回 GET / で visitor_token cookie が設定される」で扱う

### 計画外の変更が必要になった場合

テストを通すために、今回の計画項目に含まれないファイルや振る舞いの変更が必要になることがある。
その場合は実装を止めて、以下を確認する:

1. **計画の順序が間違っていないか？**
   依存関係を見直し、先にやるべき項目が他にあるなら計画の順序変更をユーザーに提案する。
2. **順序は正しいが追加変更が必要な場合:**
   「テストを通すために〈追加変更の内容〉も必要です。この変更を含めてよいですか？」と
   ユーザーに確認してから進める。

「テストを通すために必要だから」を理由に、計画外の変更を無断で行わない。

---

## Step 5: 🟢 Green — 成功を確認する

```bash
bundle exec rspec <spec_file_path>
```

パスを確認し、出力をユーザーに見せる。

**パスが確認できるまで Step 6 に進まない。**

---

## Step 6: 🔵 Refactor — 必要なら整理する

実装コードと spec の両方を見直し、読みやすくなる箇所があれば整理する。
ただしリファクタリングの対象は**今回触ったコードだけ**。他の箇所に手を広げない。

整理したら再度テストを実行してグリーンを維持していることを確認する。

リファクタリングが不要であれば、そのまま Step 7 へ進む。

---

## Step 7: 関連テストを実行する

今回の変更が既存テストを壊していないことを確認する。
まずは責務に近い最小スコープで確認し、全体実行は節目に限定する。

```bash
# 1. まず今回変更した spec ファイルだけ
bundle exec rspec <今回変更した spec ファイル>

# 2. 次に同責務のディレクトリ
bundle exec rspec <同責務の spec ディレクトリ>

# 3. 節目（計画の半分を超えた時点、または作業全体の完了直前）に全体
bundle exec rspec spec/
```

通常サイクルでは 1 → 2 まで。3 は節目に限定する。

失敗があれば修正し、グリーンを維持してから終了する。

---

## Step 8: サイクル終了 — 必ず以下を報告する

```
---
🏁 今回のサイクル完了

【今回対象にした振る舞い】
（内部実装名ではなく振る舞いの表現で記述する。例:）
（○ visitor_token がない Visitor は無効）
（○ 初回 GET / で Visitor が1件作成される）
（× visitor_token の presence バリデーション ← 振る舞い表現に言い換える）

【完了したこと】
- spec: （パス）に「（it のタイトル）」を追加
- 実装: （ファイルパス）に（何を）追加

【残っていること】
（計画のうち、まだ完了していない項目を番号付きで列挙）

【次サイクル候補メモ】
（実装中に気づいた「ついでにできそうだったが見送った項目」があればここに記録）

【作業全体の状態】
未完了 / 完了

【次にやること】
（計画の次の項目を示す）

まだ残りがあります。続けるには /tdd-cycle を再実行してください。
---
```

作業全体が完了した場合のみ「完了」と記載し、それ以外は「未完了」とする。

---

## 注意事項

- **ブランチ作成・切り替えはしない**（すでに正しいブランチにいる前提）
- **Issue 作成はしない**
- **引数は受け取らない**（コンテキストから対象を判断する）
- 1サイクルで Issue 全体を終わらせようとしない
- Red を見ずに Green に進まない、Green を見ずに Refactor に進まない — この順序は速度より優先する
- 承認済みの計画を無断で変更しない
